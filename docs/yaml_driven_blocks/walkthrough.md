# Walkthrough: YAML Driven Layout Blocks

## 実装内容の確認

本実装により、TunageMON LPテンプレートは「特定のテーマに依存したハードコーディングされたコンポーネントの集合体」から、「完全にデータ駆動（YAML）で制御される汎用レイアウトブロックの組み合わせ」へと進化しました。

### 1. `README.md` の更新
- プロジェクトの `README.md` に、新しく導入された `blocks` 配列の書き方と、利用可能な汎用レイアウトブロック（`BlockCards`, `BlockBeforeAfter`, `BlockFAQ`）のドキュメントを追加しました。
- コンポーネントが増えすぎないように、**レイアウト（見た目）ベース** で汎用化する設計思想を明記しました。

### 2. データ構造の刷新
- `content.config.ts` で `blocks` 配列のスキーマを定義しました。
- `07-theme.md` のYAML Frontmatterに `blocks` を定義し、元のハードコーディングされていた「解決策」「導入シナリオ」「よくある質問」のデータを汎用レイアウトの形式で注入しました。

### 3. コンポーネントの整理
- 役割を終えたテーマ専用のコンポーネント（`LPSolution.astro` など）を削除しました。
- 新しい汎用ブロックコンポーネント群を `web/src/components/lp-blocks/` 配下に新設し、Web上での描画を担当させました。

### 4. Typst (PDF) の適応
- `pdf/presentation.typ` 側もブロックアーキテクチャに適応させました。
- JSONとして出力された `blocks` をループ処理し、`type == "BlockCards"` であれば「3カラムのカードスライド」を生成、`type == "BlockBeforeAfter"` であれば「矢印付きのビフォーアフタースライド」を生成するように分岐処理を組み込みました。

### 5. Markdownのゴミコード除去
- Markdown本文に記載されていた「Web用の詳細テキストをここに記述します…」という不要なプレースホルダーテキストが、画面に出力されてしまう不具合（4スペースインデントによるコードブロック化）を解消するため、当該テキストを全ファイルでコメントアウトしました。

### 今後の展望（Next Steps）
現在 `07-theme.md` のMarkdown本文に残っている「導入の流れ」「比較表」「セキュリティ」などのHTMLブロックも、同様に `blocks` に書き出し、`BlockFlow` や `BlockTable` といった新たな汎用コンポーネントを作成することで、完全なYAML駆動へと移行することが推奨されます。
